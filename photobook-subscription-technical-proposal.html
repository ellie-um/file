"앱을 안 켜도 폰 속 사진이 자동으로 포토북이 된다"는 기술적으로 불가능합니다.
대신 "앱을 켜면 순식간에 포토북이 만들어진다"는 가능하고, 사용자 경험도 더 좋습니다.

01

서비스 개요

매월 자동으로 사용자의 사진을 분석하여 포토북을 구성하고, 알림으로 추천하는 구독형 서비스.

// 서비스 플로우 사용자의 한 달 사진 → AI 클러스터링 (시간/위치 기반 그룹핑) → 이미지 품질 분석 (얼굴, 구도, 선명도) → 베스트 30장 선택 → 포토북 레이아웃 자동 구성 → 미리보기 제공 → 구매 시에만 원본 업로드

고정 스펙

  • 포토북 이미지 수: 30장 고정
  • 추천 주기: 월 1회
  • 대상: 이번 달 사진 우선, 부족 시 이전 달 포함
02

백그라운드 처리의 기술적 한계

2-1. iOS 제약

제약상세영향
사진 접근 불가앱이 실행 중이 아니면 PhotoKit API로 사진첩 접근 불가능백그라운드에서 사진 스캔 자체가 불가
BGAppRefreshTask최대 30초 실행 제한, OS가 실행 시점 결정클러스터링+분석+업로드를 30초 안에 끝낼 수 없음
BGProcessingTask충전 중 + WiFi 조건에서만 실행, 수 분 가능실행 조건이 까다로워 월 1회 보장 불가
실행 빈도iOS ML이 사용자 패턴을 학습하여 실행 시점 결정"매월 1일" 같은 예약 불가
결론: iOS에서 완전 백그라운드 포토북 생성은 불가능

2-2. Android 제약

제약상세영향
Doze 모드화면 꺼진 상태에서 네트워크/CPU 사용 제한이미지 업로드·서버 통신 차단됨
WorkManager최소 반복 주기 15분, OS가 배치로 묶어 실행정확한 타이밍 보장 불가
배터리 최적화제조사별 추가 제한 (삼성 등)앱이 강제 종료될 수 있음

Android는 iOS보다 자유도가 높지만, 원본 이미지 업로드를 백그라운드에서 안정적으로 수행하기는 여전히 어렵습니다.

2-3. 원본 이미지 업로드가 핵심 병목인 이유

// 현재 편집기의 업로드 플로우 사진 1장마다: 원본 이미지 (3~15MB) + 중크기 이미지 (800px) → 서버 전송 서버에서 분석 (얼굴 감지, 특징 추출, 품질 평가) → 파일당 2~15초 소요
시나리오소요 시간백그라운드 가능 여부
30장 업로드 (WiFi)1~3분iOS: 불가 (30초 제한) / Android: 불안정
30장 업로드 (LTE)3~8분양쪽 모두 불가
100장 업로드5~25분양쪽 모두 불가
발열연속 업로드 시 기기 발열 심함OS가 앱을 강제 종료할 수 있음
핵심: 원본 이미지를 서버에 올리는 과정이 있는 한, 백그라운드 처리는 시간·안정성 모두에서 실패할 가능성이 높습니다.
03

제안하는 설계: 포어그라운드 비동기 처리

핵심 아이디어: 원본 업로드를 없앤다. 썸네일(300×300)만으로 분석·추천을 완료하고, 원본 업로드는 구매 시점으로 미룬다.

3-2. 플로우 상세

[1단계] 클러스터링 — 앱 포어그라운드, 백그라운드 스레드 • 사진첩 메타데이터 스캔 (날짜, 위치) • 이번 달 기간으로 필터링 • 시간/위치 기반 클러스터링 • 소요 시간: 1~10초 (1,000~10,000장 기준) ↓ [2단계] 이미지 분석 — 썸네일 기반, 서버 배치 처리 • 클러스터 사진 (예: 300장)을 300x300 썸네일로 리사이즈 • 썸네일을 이미지분석서버에 배치 전송 • 10건씩 병렬 배치 → 약 10초 이내 • 분석 결과: 얼굴 감지, 품질 점수, 특징 벡터 ↓ [3단계] 베스트 30장 선택 — 서버 또는 로컬 • 분석 결과 기반으로 다양성·품질·중복 제거 후 30장 선택 • 소요 시간: 1초 미만 ↓ [4단계] 포토북 레이아웃 추천 — 추천서버 • 30장의 분석 정보로 추천서버 요청 • 포토북 레이아웃 JSON 응답 • 소요 시간: 1초 이하 (30장 규모) ↓ [5단계] 미리보기 — 모바일 로컬 • 포토북 레이아웃을 모바일에서 렌더링 • 원본 이미지는 기기에 있으므로 로컬 경로로 참조 • 서버 업로드 불필요 ↓ [6단계] 구매 — 사용자 결정 시점에만 업로드 • 사용자가 "구매하기" 클릭 • 이 시점에만 30장 원본을 서버에 업로드 • 구매 의사가 있는 유저만 로딩 경험

3-3. 시간 비교

단계기존 방식 (원본 업로드 선행)제안 방식 (썸네일 분석)
이미지 업로드30장 × 2~15초 = 60~450초300×300 썸네일 × 300장 ≈ 수 초
이미지 분석업로드와 동시 진행배치 처리 ~10초
포토북 추천~1초~1초
총 소요 시간1~8분15~25초
원본 업로드이미 완료구매 시 1~3분 (30장)

3-5. 알림 전략

Scenario A — 앱을 먼저 쓰다가 완성

사용자가 앱을 열어 다른 기능 사용 → 뒷단에서 포토북 자동 생성 (15~25초) → 인앱 알림: "이번 달 포토북이 준비됐어요!" → 미리보기 유도

Scenario B — 푸시 알림으로 유입

서버에서 푸시 발송 → 사용자가 푸시 터치 → 앱 실행 → 서베이/인트로 화면으로 시간 확보 (5~10초) → 뒷단에서 생성 → 미리보기 전환

현재 시스템과의 간극

4-1. 변경이 필요한 시스템

시스템현재변경 필요영향도
네이티브 앱
(iOS/Android)		사진 선택 → 에디터에 전달자동 클러스터링 + 썸네일 생성 + 분석 요청 + 미리보기 렌더링높음

4-2. 현재 편집기 구조의 한계

// 현재 모바일 포토북 생성 과정 프론트 웹뷰 (상품 선택) → 에디터 웹뷰 진입 (직접 URL 접근 불가, 반드시 순서대로) → 에디터가 네이티브에 사진 요청 (openImagePicker) → 네이티브가 사진 정보 전달 → 에디터가 원본을 서버에 업로드 (순차, 파일당 2~15초) → 업로드 완료 후 추천 요청 → 포토북 렌더링
구조적 문제: 에디터가 주도권을 가지므로 앱이 사진을 "밀어 넣을" 수 없고, 원본 업로드가 선행 필수이며, 직접 진입이 불가합니다. 따라서 구독서비스를 기존 에디터 플로우에 끼워 넣는 것은 구조적으로 맞지 않으며, 별도의 경량 플로우가 필요합니다.
05

AI 에이전트로 한 번에 진행하기 어려운 이유

5-1. 시스템 복잡도

이 프로젝트는 최소 4개 시스템을 동시에 수정해야 합니다. 각 시스템이 서로 다른 언어, 프레임워크, 통신 방식, 배포 방식을 사용합니다.

영역기술 스택
언어Swift, Kotlin/Java, JavaScript, Python
프레임워크UIKit/RxSwift, Android Jetpack, React/Redux, Flask/FastAPI
통신 방식postMessage (웹↔네이티브), REST API (클라이언트↔서버), Redis (서버↔서버)
배포 방식App Store/Play Store, 웹 배포, AWS Lambda/ECS

5-2. AI 에이전트의 한계

제약설명
실기기 테스트 불가iOS/Android 코드는 실기기나 시뮬레이터에서만 동작 확인 가능. AI가 빌드·실행·디버깅 사이클을 자체 수행할 수 없음
WebView ↔ Native 디버깅postMessage 기반 통신은 양쪽을 동시에 실행해야 테스트 가능
레거시 코드 이해수년간 쌓인 레거시. 코드만으로 파악하기 어려운 암묵적 규칙과 사이드이펙트
배포 파이프라인App Store 심사, 웹 배포, 서버 배포가 각각 다른 프로세스

5-3. 특히 어려운 부분

  • 네이티브 ↔ 웹뷰 경계: 프로젝트의 핵심이자 가장 디버깅하기 어려운 지점. 실제로 앱에서 웹뷰가 열리고 메시지가 오가는 것을 확인하려면 반드시 사람이 기기에서 테스트해야 합니다.
  • 상태 동기화: 클러스터링 → 분석 → 추천 → 미리보기 → 구매까지 여러 시스템에 걸쳐 상태가 전달됩니다. 한 곳에서 데이터 형식이 바뀌면 체인 전체가 영향받습니다.
  • 점진적 배포 불가: 최소한의 동작 단위가 크기 때문에 부분 배포가 어렵습니다.
06

분업 분석

옵션 A: 1명 (풀스택)

  • 전체 맥락을 한 사람이 파악 → 일관성 높음
  • 소통 비용 0
  • 시스템 간 인터페이스 불일치 위험 낮음
  • 점진적 개발 가능

적합: 프로토타입 단계, 빠른 방향 검증

옵션 B: 2명 (모바일 + AI)

  • 병렬 작업 가능 → 전체 기간 단축
  • 각자 전문 영역에 집중 가능
  • 한 명이 막혀도 다른 쪽 진행 가능
  • AI 분석 개선과 모바일 UI 동시 진행

적합: 본개발 단계, 일정 압박 시 (현재 프로젝트에 권장)

구분1명 (풀스택)2명 (모바일 + AI)
소통 비용없음높음 (매일 싱크 필요)
속도느리지만 일관적빠르지만 통합에 시간 소요
핵심 리스크기간이 2~3배 길어짐인터페이스 불일치 시 양쪽 재작업
전제 조건—API 스펙을 개발 전에 확정해야 함
권장: 2명 분업을 권장하되, 시작 전에 반드시 API 인터페이스 문서를 먼저 확정하고, 주 2회 이상 통합 테스트를 진행해야 합니다. 1명이 할 수 없는 규모는 아니지만, 기간이 2~3배 길어질 수 있습니다.

핵심 메시지

01

"앱이 꺼져 있어도 자동으로 다 된다"는 기술적으로 불가능합니다 — iOS 사진 접근 정책과 백그라운드 시간 제한 때문

02

"앱을 켜면 순식간에 완성된다"는 가능합니다 — 원본 업로드를 제거하고 썸네일 분석으로 전환하면 15~25초에 완료

03

4개 이상의 시스템을 동시에 수정해야 합니다 — AI 에이전트만으로는 실기기 테스트가 불가능하여 한계가 있음

04

2명 분업이 현실적이나, API 인터페이스 사전 확정과 주기적 통합 테스트가 전제 조건입니다
